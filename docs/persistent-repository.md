# Persistent Repository

Global Impact Catalyst uses a durable SQLite reference repository without changing the canonical v1.1.0 impact-contract shape or calculation rules.

## Storage model

The repository stores each canonical contract as a complete JSON snapshot and builds indexed projections for operational queries. The snapshot remains the source of truth. Projections make workspaces, initiatives, goals, indicators, observations, targets, and sources searchable without fragmenting the calculation record.

```text
canonical contract snapshot
        ├── workspace and initiative projections
        ├── goals, indicators, observations, targets, and sources
        ├── evidence repository and provenance graph
        └── indicator registry bindings and governed definitions
```

The repository also stores portfolios, portfolio memberships, draft autosaves, import receipts, restore receipts, append-only audit records, source versions, evidence items, datasets, units, indicator definitions, baseline models, target models, methods, and their immutable versions.

## Shared application service

`python/global_impact_service.py` contains use cases shared by the CLI and application wrapper:

- create and explicitly save initiatives;
- autosave drafts against a saved revision;
- list, search, duplicate, archive, and restore initiatives;
- import v1.0.x records and v1.1.0 contracts;
- manage evidence and provenance;
- register units and measurement definitions;
- export evidence chains, indicator registries, and workspace bundles.

Interfaces should call this service rather than duplicate migration, validation, contract-building, evidence, or registry rules.

## Optimistic concurrency

Every stored contract and mutable registry entity has an integer revision. Updates may provide an expected revision. A stale update raises `OptimisticConcurrencyError` with the expected and current revisions. WordPress returns the equivalent conflict response.

This prevents a background autosave, older browser tab, or outdated registry editor from silently overwriting a newer saved record.

## Save states

- `draft autosave` is recoverable working state tied to a base saved revision.
- `saved` is the explicit durable contract revision.
- Saving a contract clears its autosave.
- Autosave never increments the explicit contract revision.

## Calculation integrity

Persistence stores the entire contract generated by the canonical engine. Reopening, exporting, importing, and restoring a contract do not recalculate or alter derived metrics. Editing compact authoring facts and explicitly saving creates a new contract snapshot through the same canonical engine.

## Evidence repository extension

Schema 4 adds source, source-version, evidence, dataset, provenance, and claim-link records. These remain additive to immutable canonical contract snapshots.

## Indicator registry extension

Schema 5 adds units, indicator definitions, baseline models, target models, method definitions, immutable versions, and stable registry bindings. Registry materialization records how an indicator is defined and governed; it does not silently modify the underlying contract or claim eligibility.
